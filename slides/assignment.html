PART 1
Which strategy fits your team?
In your room, talk it through: no writing, just the argument. The three strategies from the teaching deck meet an actual team's reality.
1
Go around the room. Each person sketches their own team or project out loud: release cadence · regulatory constraints · team size · how often production ships.
2
Argue the fit. For each person, the room agrees which of the three strategies fits their reality, and why. Disagreement is the point; talk it out.
3
Land on one for the room. You'll run Part 2 in Git Flow regardless, but knowing which one you'd actually pick is what makes feeling the difference land.
  • Git Flow: parallel supported versions, release + hotfix branches, every hotfix merges twice.
  • GitHub Flow: one main, everything returns through a PR, merging is releasing.
  • Trunk-based: branches live hours, feature flags hide unfinished work, needs serious CI discipline.
No team?

Never worked on a multi-engineer team? Argue it for an Ignition team you've consulted with, or use the scenario in docs/branching-strategies.md as a stand-in.

Part 2 · Git Flow · for real
The release-week scenario, but this time it's you

The same story from the teaching deck. v1.2 is live in production. You start shipping the v2.0 feature, and mid-work, a P1 bug in production ambushes you. You thread both through Git Flow, exactly the way Chapter 1 drew it.

  • Step 1, set the stage: main + develop, v1.2 tagged as what's live.
  • Step 2, the v2.0 feature: add a new Perspective view on a feature/* branch → PR.
  • Step 3, the ambush: a P1 bug in production. Fix it on a hotfix/* branch, merged Git Flow's way, twice.
Mustry Academy · CI/CD for IgnitionPart 2 · steps 1 → 3
Part 2 · step 1 · set the stage
STEP 1
Set up the Git Flow world
Two long-lived branches and a tag marking what's in production. This is the graph the scenario runs on.
1
On your fork, make sure main exists and create develop off it: the two long-lived branches Git Flow needs.
2
Tag what's live: v1.2 on main. This is "production", the thing the P1 bug is running against.
3
Push both the branch and the tag to your fork. Now the world matches the teaching-deck diagram.
# two long-lived branches
git switch main
git switch -c develop
git push -u origin develop

# mark what's live in production
git switch main
git tag v1.2
git push origin v1.2      # → "production" is v1.2
The mental model

main = production · develop = the next release in progress · v1.2 = the exact commit customers are running. Every branch you cut next answers "off which of these?"

Part 2 · step 2 · the v2.0 feature
STEP 2
Ship the v2.0 feature: a new Perspective view
A small, real Ignition change: add a view so it goes live in the project. Off develop, the Git Flow way.
1
Branch off develop (not main, because features target the next release): feature/v2-<view-name>-<ini>.
2
Add a new view. Easiest path: copy the overview resource folder to a new name under views/pages/, edit its view.json title, and register the page in page-config/config.json so it's reachable.
3
Validate → commit → PR into develop. scripts/validate.sh green, Conventional Commits message, template filled in, link in chat. Leave it open: the ambush comes before you merge.
git switch develop
git switch -c feature/v2-status-<ini>

# add a new Perspective view (copy overview → rename)
#   projects/lab-project/…/views/pages/status/
#   register it in page-config/config.json

scripts/validate.sh          # → green
git add -A
git commit -m "feat(views): add v2.0 status page"
git push -u origin feature/v2-status-<ini>
# PR: base = develop · link in chat · don't merge yet
See it live?

Gateway up? scripts/scan.sh (or docker compose restart), then open the project: your new page is there. That's the "it goes live" moment.

Part 2 · step 3 · the P1 ambush
STEP 3
P1 in production: hotfix it the Git Flow way
Your feature is still open when a customer hits the null-reading crash in v1.2, live. It can't wait for v2.0. This is where Git Flow earns its keep.
1
Branch the hotfix off production (main / the v1.2 tag, not develop): hotfix/null-reading-<ini>. Your v2.0 work isn't in production, so it must not ride along.
!
The bug: lab.display.format_reading() does "%.1f" % None on a null reading (comms loss) and throws. Fix: a placeholder like "-- °C" for None / bad input, while keeping 0 formatting as "0.0 °C".
2
Merge it twice, the Git Flow move: into main (production gets the fix), and into develop (so v2.0 doesn't regress it). Tag v1.2.1 on main.
3
Now finish the feature. Pull the hotfix down into your feature branch, then its PR into develop is clean. Two PRs, your peers review both.
# hotfix off PRODUCTION, not develop
git switch main
git switch -c hotfix/null-reading-<ini>

# fix format_reading() in lab/display/code.py
scripts/validate.sh
git commit -am "fix(display): placeholder for null readings"

# the Git Flow double-merge (via PRs)
#   PR 1: hotfix → main   → tag v1.2.1
#   PR 2: hotfix → develop

# then update the feature and open ITS PR → develop
git switch feature/v2-status-<ini>
git merge develop        # pick up the hotfix
The whole lesson

A hotfix that only lands on main reappears next release, because develop never got it. Git Flow's "merge twice" is annoying and exactly why it exists. You just felt both.

Part 3 · the review
PART 3 · A
Review a peer's PR
You each have a hotfix PR and a feature PR open. Coordinate in chat so each PR gets one reviewer.
1
Pick a peer's PR from the cohort chat; the hotfix is the richest to review.
2
Read slowly (at least 5 minutes) before writing anything. Hotfix: do valid readings still format (-6.5 → "-6.5 °C", and 0 → "0.0 °C")? And, the Git Flow check, did it branch off production, not develop? Feature PR: JSON well-formed, page registered, change scoped?
3
Check their verification: did they run validate.sh? Could you actually follow their "How to test"? Read the description and commit message too.
4
Comment: at minimum a praise: and a suggestion:, with an issue: only if something is actually wrong. No quota; quality over quantity.
5
Verdict: Request changes if you left at least one issue:, else Approve.
# a review that helps: three real comments
praise: tight diff, one function, easy to review

suggestion: lab.util.to_float already swallows
  None and junk input, so reuse it here?

issue: format_reading(0, "°C") now returns
  "-- °C", but zero is a valid reading.
Tone

Labels don't make rude feedback polite. If you wouldn't say it in person, don't write it in the PR. "?" is not a review comment.

Part 3 · respond & merge
PART 3 · B
Respond, then merge, Git Flow's way
Close the loop on all three branches, in the right order: hotfix first (production is on fire), then the feature.
1
On your PRs, react to every comment: +1 the praise, answer questions, push a fix-up commit for what you accept, and decline with reasoning what you don't.
2
Additive commits, no force-push during review. Re-request review when you're ready.
3
Merge in Git Flow order: the two hotfix PRs first (→ main, tag v1.2.1, then → develop), then the feature PR into develop. Delete each branch after merge.
# a fix-up during review: additive, never force-push
git add -p
git commit -m "fix(display): reuse lab.util.to_float"
git push

# merge order matters in Git Flow:
#  1. hotfix → main      → tag v1.2.1  (prod fixed)
#  2. hotfix → develop               (no regression)
#  3. feature → develop              (v2.0 progresses)
The payoff

You didn't read about the double-merge; you did it, under a fake fire drill. That's the difference between knowing Git Flow and having felt it.

Part 3 · contribute upstream
PART 3 · C
Send one PR back to the upstream repo
Every PR so far had base = your fork. Now open one with base = the upstream repo: that's exactly how open source works.
1
Pick your cleanest branch. The hotfix is ideal: a small, real fix any maintainer would want.
2
Open a PR with base = mustry-academy/cicd-lab-02-… (the upstream), head = your fork's branch. GitHub calls this a cross-fork PR.
3
Write it for a stranger. The maintainer doesn't know your context: the What / Why / How to test is all they get. That's why the template matters.
This is open source

Fork → branch → PR back to the original repo is the entire open-source contribution model. Linux, Python, Ignition modules on GitHub: all of it is this loop, at scale. You just did it once.

We won't merge it

We're not merging cohort PRs into the course repo; the point is doing the cross-fork PR, not landing it. Maintainers get dozens of these; a clean, self-describing one is how yours gets noticed.

Optional stretch
OPTIONAL · 1
Stretch 1: now do it in GitHub Flow
Same scenario, one long-lived branch. The point isn't the code; it's feeling what Git Flow's ceremony bought you, and what it cost.
1
Reset your head to GitHub Flow: one main, no develop, no release/*. Every branch is short-lived and comes back through a PR into main.
2
Run the same two changes: the v2.0 view on a feature/* branch, the null-reading fix on a fix/* branch, and both PR straight into main.
3
Notice the difference. The hotfix merges once, not twice. But where does v1.2 live? If prod were an older release, GitHub Flow has nowhere to put the fix. That's the whole trade-off, in your hands.
# GitHub Flow: everything targets main
git switch main
git switch -c feature/v2-status-<ini>   # → PR into main

git switch main
git switch -c fix/null-reading-<ini>    # → PR into main

# one merge each. no develop, no double-merge.
# simpler, until more than one version is live.
The takeaway

Git Flow's double-merge felt like bureaucracy, until you ask GitHub Flow to patch an old version and it can't. You now know both from the inside.
