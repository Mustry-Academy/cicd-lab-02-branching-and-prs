Part 2 · GitHub Flow · for real
The release-week scenario, but this time it's you

The same story from the teaching deck. v1.2 is live in production. You start shipping the v2.0 feature, and mid-work, a P1 bug in production ambushes you. You thread both through GitHub Flow, exactly the way Chapter 1 drew it.

  • Step 1, set the stage: one main, v1.2 tagged as what's live.
  • Step 2, the v2.0 feature: add a new Perspective view on a feature/* branch → PR.
  • Step 3, the ambush: a P1 bug in production. Fix it on a fix/* branch — one PR into main, released with the v1.2.1 tag.
Mustry Academy · CI/CD for IgnitionPart 2 · steps 1 → 3
Part 2 · step 1 · set the stage
STEP 1
Set the GitHub Flow stage
One long-lived branch and a tag marking what's in production. This is the graph the scenario runs on.
1
GitHub Flow needs just one long-lived branch: main. Every branch you cut starts there and returns there through a PR.
2
Tag what's live: v1.2 on main. This is "production", the thing the P1 bug is running against. In GitHub Flow, tags carry the release history.
3
Push the tag to your fork — your first real git push of the course, named this morning, used now. The world matches the teaching-deck diagram.
# one long-lived branch: main is it
git switch main

# mark what's live in production
git tag v1.2
git push origin v1.2      # → "production" is v1.2
New today

git tag pins a permanent name to one exact commit; unlike a branch, it never moves — that's what makes it a release marker. git push uploads your local commits and refs to a remote (origin = your fork). Tags don't ride along with a normal push — git push origin v1.2 sends one explicitly — and a new branch needs git push -u origin <branch> once; the -u links it to your fork, so later pushes are plain git push.

The mental model

main = the one shared line of development · v1.2 = the exact commit customers are running · releasing = merging to main and tagging. Every branch you cut next is short-lived: off main, back to main.

Part 2 · step 2 · the v2.0 feature
STEP 2
Ship the v2.0 feature: the silo detail view
Oatmakers' v2.0 scope starts with a silo detail view. A small, real Ignition change: add it so it goes live in the project. Off main, the GitHub Flow way.
1
Branch off main — in GitHub Flow every branch starts at main and returns to it through a PR: feature/v2-silo-detail-<ini>.
2
Add the view. Easiest path: copy the overview resource folder to silo-detail under views/pages/, edit its view.json title, and register the page in page-config/config.json so it's reachable.
3
Validate → commit. scripts/validate.sh green, then commit in Conventional Commits style: type(scope): summary, where the type names the kind of change — today you'll need feat and fix. Machines can parse it too, which pays off when releases get automated.
4
PR into main. Template filled in, link in chat, leave it open: the ambush comes before you merge. (A copied view makes a big diff; that's fine — the review focuses on the page registration and valid JSON, not the line count.)
git switch main
git switch -c feature/v2-silo-detail-<ini>

# add the silo detail view (copy overview → rename)
#   projects/lab-project/…/views/pages/silo-detail/
#   register route /silo-detail in page-config/config.json

scripts/validate.sh          # → green
git add -A
git commit -m "feat(views): add v2.0 silo detail view"
git push -u origin feature/v2-silo-detail-<ini>
# PR: base = main · link in chat · don't merge yet
See it live?

Gateway up? scripts/scan.sh (or docker compose restart), then open the project: your new page is there. That's the "it goes live" moment.

Part 2 · step 3 · the P1 ambush
STEP 3
P1 in production: fix it the GitHub Flow way
Your feature is still open when a customer hits the null-reading crash in v1.2, live. It can't wait for v2.0. This is where merge order starts to matter.
1
Branch the fix off main: fix/null-reading-<ini>. Because your feature PR is still open, main is v1.2 — the fix carries no v2.0 work.
!
The bug: lab.display.format_reading() does "%.1f" % None on a null reading (comms loss) and throws. Fix: a placeholder like "-- °C" for None / bad input, while keeping 0 formatting as "0.0 °C".
2
One PR into main. That's the whole GitHub Flow ceremony: no develop, no double-merge. Don't merge yet: reviews come first; Part 3 merges in order and tags v1.2.1.
3
That's two PRs open (feature → main, fix → main): both links in chat, a peer on each. Merge order is the lesson: the fix lands (and v1.2.1 is tagged) before the feature merges — in GitHub Flow, merging is releasing.
# the fix branches off main — which IS v1.2,
# because the feature hasn't merged yet
git switch main
git switch -c fix/null-reading-<ini>

# fix format_reading() in lab/display/code.py
scripts/validate.sh
git commit -am "fix(display): placeholder for null readings"
git push -u origin fix/null-reading-<ini>

# ONE PR: fix → main
#   reviewed → merged FIRST → tagged v1.2.1  (Part 3)
The whole lesson

Merging is releasing: whatever is on main when you tag v1.2.1 ships in v1.2.1. Merge the feature first and your "patch" quietly ships half of v2.0. Order is the one ceremony GitHub Flow kept.

Part 3 · the review
PART 3 · A
Review a peer's PR
You each have a fix PR and a feature PR open. Coordinate in chat so each PR gets one reviewer.
1
Pick a peer's PR from the cohort chat; the fix is the richest to review.
2
Read slowly (at least 5 minutes) before writing anything. Fix PR: do valid readings still format (162.0 → "162.0 °C", and 0 → "0.0 °C")? And, the GitHub Flow check: is the fix just the fix (no v2.0 work riding along), set to merge before the feature? Merging is releasing. Feature PR: JSON well-formed, page registered, change scoped?
3
Check their verification: did they run validate.sh? Could you actually follow their "How to test"? Read the description and commit message too.
4
Comment: at minimum a praise: and a suggestion:, with an issue: only if something is actually wrong. No quota; quality over quantity.
5
Verdict: Request changes if you left at least one issue:, else Approve.
# a review that helps: three real comments
praise: tight diff, one function, easy to review

suggestion: lab.util.to_float(value, default=None)
  swallows junk input for you — keep the None
  check, though, or nulls format as "0.0 °C".

issue: format_reading(0, "°C") now returns
  "-- °C", but zero is a valid reading.
Tone

Labels don't make rude feedback polite. If you wouldn't say it in person, don't write it in the PR. "?" is not a review comment.

Part 3 · respond & merge
PART 3 · B
Respond, then merge, in release order
Close the loop on both branches, in the right order: the fix first (production is on fire), then the feature.
1
On your PRs, react to every comment: +1 the praise, answer questions, push a fix-up commit for what you accept, and decline with reasoning what you don't.
2
Additive commits, no force-push during review. Re-request review when you're ready.
3
Merge the fix PR first (production's on fire), then tag v1.2.1 on main — in GitHub Flow, the tag is the release. Keep the fix branch: Part C sends it upstream.
4
Now finish the feature: pull the fix down (git merge main), push, then merge its PR into main and delete the feature branch.
# a fix-up during review: additive, never force-push
git add -p
git commit -m "fix(display): reuse lab.util.to_float"
git push

# merge order = release order in GitHub Flow:
#  1. fix → main, then tag the release
git switch main && git pull
git tag v1.2.1 && git push origin v1.2.1

#  2. update the feature, then feature → main
git switch feature/v2-silo-detail-<ini>
git merge main            # pick up the fix
git push
The payoff

You didn't read about merge order; you released under a fake fire drill — fix, tag, then feature. That's GitHub Flow handling a release week.

Part 3 · contribute upstream
PART 3 · C
Send one PR back to the upstream repo
Every PR so far had base = your fork. Now open one with base = the upstream repo: that's exactly how open source works.
1
Pick your cleanest branch. The fix is ideal: a small, real fix any maintainer would want. (Already deleted it? The merged PR's page has a Restore branch button — or just push the branch again.)
2
Open a PR with base = mustry-academy/cicd-lab-02-… (the upstream), head = your fork's branch. GitHub calls this a cross-fork PR — and this once, the upstream default you've been switching away from is exactly right.
3
Write it for a stranger. The maintainer doesn't know your context: the What / Why / How to test is all they get. That's why the template matters.
This is open source

Fork → branch → PR back to the original repo is the entire open-source contribution model. Linux, Python, Ignition modules on GitHub: all of it is this loop, at scale. You just did it once.

We won't merge it

We're not merging cohort PRs into the course repo; the point is doing the cross-fork PR, not landing it. Maintainers get dozens of these; a clean, self-describing one is how yours gets noticed.

Optional stretch
OPTIONAL · 1
Stretch 1: now do it in Git Flow
Same scenario, the heavyweight strategy. The point isn't the code; it's feeling what Git Flow's ceremony buys — support for more than one live version — and what it costs.
1
Rewind the repo first. After Part 2, main already has both changes: reset your fork's main back to v1.2 and clear the local feature branch (it's your throwaway fork; force-push is fine here — and note it's exactly what Stretch 2's protection forbids on a shared repo).
2
Build the Git Flow world: a long-lived develop next to main. Rerun the two changes Git Flow's way: the view on feature/* off develop → PR into develop; the fix on hotfix/* off main → two PRs, into main (tag v1.2.1) and into develop.
3
Feel the difference. The hotfix merges twice — skip the develop merge and v2.0 regresses the fix next release. In Part 2, v1.2 lived in a tag; here a whole branch structure guards it. That's the trade-off, in your hands.
# rewind the fork to before the scenario
git switch main
git reset --hard v1.2
git push --force origin main
git branch -D feature/v2-silo-detail-<ini>  # local leftover
git tag -d v1.2.1
git push origin --delete v1.2.1         # un-release the patch

# the Git Flow world: TWO long-lived branches
git switch -c develop
git push -u origin develop

# same two changes, Git Flow's way:
#   feature/v2-… off develop → PR into develop
#   hotfix/null-… off main   → TWO PRs:
#     → main (tag v1.2.1)  AND  → develop
# the hotfix merges TWICE — that's the point
The takeaway

GitHub Flow answered release week with one merge and a tag. Git Flow's double-merge is what it costs to support old versions properly. You now know both from the inside.
