Branching strategies · choosing
03
Picking one — the decision matrix
There is no "best" strategy. There's the one that fits your team's reality — release cadence, versions in production, and discipline level.
Your situationBest fit
Single SaaS product, continuous deploy, 2–10 engineersGitHub Flow
Multiple released versions live at once — you still patch the old ones (e.g. a v1.2 hotfix while v2.0 ships)Git Flow (release/hotfix branches give each version a place to live)
Strict regulatory environment, lengthy validation per releaseGit Flow (release branches make audit easy)
Feature flags and fast, trusted CI already in place, and you want to integrate continuously to mainTrunk-based (needs both — it's GitHub Flow with the branch life cut to hours)
Brand-new team, learning Git for the first timeGitHub Flow (lowest cognitive overhead)
Ignition team deploying one project to many gateways (sites, lines, or client installs)GitHub Flow — one tag per release
The real question is just one

Six rows, but the split is binary: is more than one version live at a time? No → GitHub Flow (trunk-based is its high-discipline extreme). Yes → Git Flow. Everything else is a variation. This course uses GitHub Flow for the labs. For your team, pick what fits your reality — not the one that sounds most sophisticated. You'll write that choice down in the you-do.

Pull requests · reviewability
06
What makes a PR easy to review
Four properties, in order of importance. The template enforces three of them by structure — smallness is on you.
1 · Small
  • Under 200 lines of diff, ideally under 50
  • Reviewers find more bugs in small PRs — they actually read them
  • A 1,000-line PR gets rubber-stamped, no matter how rigorous the reviewer feels
2 · One concern
  • A bugfix is not a refactor
  • A rename is not a feature
  • A formatting pass is not a code change
3 · Self-describing
  • What does this change? (one sentence)
  • Why now? (the bug, ticket, or request)
  • How does a reviewer verify it? (commands to run)
4 · Tested
  • A test that would have caught the bug, or…
  • …an explicit manual test plan in "How to test"
  • In this lab: ops/validate.sh + the behavior you changed
deep-dive docs/pr-review-style.md — the fuller toolkit (optional)

Pull requests · review feedback
07
Leaving feedback that helps
Prefix every comment with what kind it is, so the author knows how to react. Three labels cover almost everything today.
LabelMeaningExample
praise: Something genuinely good — tells the author what's working "praise: nicely focused diff, easy to review."
suggestion: A specific, optional improvement — accept, decline with reasoning, or discuss "suggestion: collapse these two ifs into one."
issue: Something that should change before merge "issue: this still throws on a null reading; format_reading(None, '°C') errors instead of showing a placeholder."
nitpick: question: thought: chore: The fuller set, when you want it — see docs/pr-review-style.md · conventionalcomments.org
Tone

Labels don't make rude feedback polite. If you wouldn't say it in person, don't write it in the PR. Async amplifies tone — neutral in your head reads cold on screen. "?" is not a review comment.

Pull requests · verdict & merge
08
The verdict — and the merge button
A review ends in one of three verdicts; an approved PR ends in one of three merges. Both are choices your team agreement should make for you.
Three verdicts
  • Approve — happy to merge as-is; trivial nitpicks are fine inside an approve
  • Request changes — there's at least one issue: the author must address
  • Comment-only — you read it but aren't voting (not your area, or someone already approved)
  • Common mistake: request-changes for nitpicks — friction without value
Three merges (GitHub's dropdown)
  • Merge commit (--no-ff) — preserves the branch shape in the graph
  • Squash & merge — the whole PR becomes one commit on main
  • Rebase & merge — replays commits linearly, no merge commit
  • Pick the one your agreement specified — that's the whole point of writing it
Authors have rights too

Review is a negotiation. Respond to every comment (+1, a reply, or a fix-up commit) — but declining a suggestion with reasoning is a valid response. And don't force-push during review: it nukes the reviewer's context. Additive commits now, squash on merge if you must.

Ignition · the project on disk
09
An Ignition 8.3 project is a folder of small text files
This is the real project from this lab's repo (projects/lab-project/) — a Perspective HMI screen and two script libraries. What the Designer edits is what Git tracks.
projects/lab-project/
├── project.json                     ← project manifest: title, enabled, parent
├── com.inductiveautomation.perspective/   ← one folder per module
│   ├── views/pages/overview/        ← one folder per resource
│   │   ├── view.json                ← the payload: the whole view
│   │   └── resource.json            ← the manifest: scope, files list
│   ├── page-config/  config.json + resource.json
│   └── session-props/  props.json + resource.json
└── ignition/script-python/lab/
    ├── display/  code.py + resource.json  ← lab.display script library
    └── util/     code.py + resource.json  ← lab.util helpers
  • Resources are folders. Each one is a resource.json manifest plus its payload file(s): view.json for a Perspective view, code.py for a script library.
  • Namespaced by module. com.inductiveautomation.perspective, ignition/script-python — each module owns its subtree.
  • It's all JSON and Python. Which means: diffable, blame-able, branchable, reviewable — everything from this morning applies.
  • In this lab the compose stack bind-mounts only ./projects into the gateway — the file you edit on your laptop is the file the gateway runs. Gateway config stays in a named volume, out of the repo (that story is Lab 04's).

Ignition · what belongs in the repo
10
What to commit vs what to .gitignore
Rule of thumb: if a human authored it, commit it. If the gateway (re)generates it, ignore it.
Commit — human-authored
  • project.json — the project manifest
  • resource.json + payloads (view.json, code.py, config.json)
  • .gitignore / .gitattributes / PR template — the repo's own rules
  • docker-compose.yml, ops/ scripts, docs
Ignore — generated or secret
  • **/.resources/ — the gateway's blob store, regenerated on boot/scan
  • **/*.digest.json, **/.metadata/, migration-log-*.md — gateway churn
  • .env — credentials never enter history (commit .env.example instead)
  • OS/editor junk — .DS_Store, .idea/, __pycache__/
The line-ending trap

This repo's .gitattributes forces LF on all text files. Without it, Windows clones sprinkle CRLF into Ignition's JSON — noisy diffs on every file you never touched, and it can confuse the gateway's project-scan heuristics. If gateway-generated files ever show up in git status, your ignore rules are off — fix that before committing.

Ignition · saves become diffs
11
A Designer save is a file write — so it's a diff
Save a view in the Designer → the gateway writes these JSON files. Change one label → a one-line diff a reviewer reads in ten seconds. That's what makes PR-sized Ignition changes possible.
  • The chain: Designer save → project files on the gateway → git status shows exactly what changed → git add -p → a scoped commit → a small PR.
  • In this lab your editor plays the Designer — same files, same diffs, The bind mount means that changes are bidirectional, make a change in the designer, and you'll see it in your files, make a change in the files, and you'll see it in the designer.
  • Read your own diff before you commit. It's the cheapest review you'll ever get — and the habit that keeps Ignition PRs small.
# git diff after changing one label text
--- a/…/views/pages/overview/view.json
+++ b/…/views/pages/overview/view.json
   "meta": { "name": "Subtitle" },
   "props": {
-    "text": "Plant 4 — Refrigeration Overview",
+    "text": "Plant 4 — Refrigeration Overview (night shift)",
     "style": {
       "color": "#6b7f9c",

# and for a script change — plain Python diffs
--- a/…/script-python/lab/display/code.py
+++ b/…/script-python/lab/display/code.py
 def format_reading(value, units):
+    if value is None:
+        return "%s %s" % (PLACEHOLDER, units)

Ignition · editing outside the Designer
12
You can also edit straight in VS Code
The project is just text on disk, so your editor is a first-class way in — but the gateway won't notice your edits until you tell it to scan.
  • Recommended VS Code extensions for this course:
Extensions
  • Flint for Ignition — Ignition project support in the editor
  • Docker — manage the lab's containers without leaving VS Code
  • Remote Development extension pack — edit inside the container / over SSH as if it were local
The catch

Edits made through VS Code are not picked up automatically by the running gateway. You have to trigger a project scan (for project resources) and a config scan (for gateway config files) via the gateway's API.

# edit a file in VS Code…
$ code projects/lab-project/.../code.py

# …then tell the gateway to pick it up:
$ ops/scan.sh
→ POST /data/api/v1/scan/projects   # project resources
→ POST /data/api/v1/scan/config     # gateway config files
   header: X-Ignition-API-Token: <key>
✓ gateway now serves your edit
  • Two scans, two jobs. A project scan reloads views & scripts; a config scan reloads gateway config. VS Code edits usually need both.
  • Or install a module. The ignition-project-scan-endpoint module adds one clean endpoint — POST /data/project-scan-endpoint/scan — that a script (and later, CI) can call to reload the project.

Ignition · when branches collide
13
Why Ignition merge conflicts are special
Git merges lines. Ignition resources are structures. That mismatch is where teams get burned.
  • One view = one JSON document. Two people editing the same view — even different components — often touch neighbouring lines of the same file. Conflicts happen far more easily than in code split across many files.
  • Conflict markers break the JSON. A half-resolved view.json is invalid — the gateway can't load it. Run ops/validate.sh after every resolve; red output is your safety net.
  • A "clean" auto-merge can still be wrong. Git happily merges two edits to the same children array into JSON that parses but renders nonsense. Valid ≠ correct — open the view after merging.
  • Binary resources don't merge at all. Images, .gwbk backups, .modl files — it's one side or the other. Coordinate, don't merge.
# mid-merge view.json — this is NOT valid JSON
      "props": {
<<<<<<< HEAD
        "text": "Discharge Temp",
=======
        "text": "Discharge Temperature",
>>>>>>> feature/rename-tiles
        "style": { … }
The defence is chapter 1 + 2

Short-lived branches and small PRs shrink the collision window; one owner per view at a time removes it. The strategy you write in the you-do is your team's conflict insurance — and hotfixing live in the Designer bypasses Git entirely, which is how "who changed the plant overview?" mysteries are born.
